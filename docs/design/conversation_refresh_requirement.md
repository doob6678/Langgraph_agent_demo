# 会话刷新加载与记忆存储需求

## 1. 目标

- 页面每次刷新时，自动加载当前用户最近 5 条对话，恢复可读上下文。
- 明确存储边界：MySQL 仅用于会话对话记录；长期记忆统一写入 Milvus。

## 2. 范围

- 前端：页面初始化时触发“最近会话加载”请求并渲染。
- 后端：提供最近会话读取接口，按用户维度隔离数据。
- 记忆：长期记忆取消混合存储，单条内容长度限制为 1024。

## 3. 功能需求

### 3.1 页面刷新自动加载最近 5 条

- 触发时机：页面初始化完成后立即触发一次加载。
- 加载条数：固定最近 5 条，按时间倒序读取、正序渲染。
- 隔离规则：仅返回当前 `user_id` 的会话内容。
- 失败处理：接口失败时不阻断主聊天流程，前端展示轻量错误提示并允许继续对话。

### 3.2 后端接口约定

- 建议接口：`GET /api/conversations/recent?user_id={user_id}&limit=5`
- 参数默认值：`limit` 默认 5，最小 1，最大 20。
- 返回结构建议：
  - `conversations`: 最近会话数组（每条包含 `id`、`role`、`content`、`created_at`）
  - `total`: 本次返回条数

## 4. 存储设计约束

### 4.1 短期会话存储

- 使用 MySQL 存储短期对话（会话上下文），支撑刷新恢复与窗口管理。

### 4.2 长期记忆存储

- 长期记忆仅写入 Milvus。
- 不使用 MySQL 长文本扩展表参与长期记忆读写。
- 不使用混合存储方案。
- 单条长期记忆内容超出 1024 字符时，写入前截断为前 1024 字符。

## 5. 验收标准

- 刷新页面后可见最近 5 条对话，顺序正确。
- 不同用户之间数据不可见。
- 新增长期记忆内容长度严格不超过 1024。
- 长期记忆新增、检索、删除流程不依赖 MySQL 长文本表。
