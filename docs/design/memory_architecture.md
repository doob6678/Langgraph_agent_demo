# 记忆模块架构设计与改造方案

## 1. 短期记忆 (Context) 管理

### 现状痛点
目前的 `MemoryManager` 和 `AgentState` 缺乏显式的滑动窗口机制。这导致在长对话中，短期记忆 (`memory_context`) 可能无限增长，最终超出 LLM 的上下文窗口限制或导致性能下降。

### 改造方案
引入可配置的滑动窗口机制来管理会话历史。

#### 配置项
- **参数名**: `MAX_SHORT_TERM_WINDOW`
- **位置**: 环境变量 (`.env`) 或系统配置。
- **默认值**: `10` (保留最近 10 轮对话)。

#### 实现逻辑
1.  **加载**: `MemoryManager` 初始化时读取环境变量 `MAX_SHORT_TERM_WINDOW`。
2.  **截断**: 在追加新消息前，检查当前列表长度。
3.  **FIFO 策略**: 若超出限制，丢弃最旧的消息，仅保留最近 N 轮。
4.  **线程安全**: 操作在线程安全上下文中执行（必要时加锁），防止并发请求导致的数据竞争。

## 2. 主动长期记忆 (Facts)

### 现状痛点
目前大模型只能在 System Prompt 中**被动**接收记忆。它缺乏“主观能动性”去**主动**保存从用户输入中提取的新事实（例如“我是Python开发者”）。

### 改造方案
赋予 LLM 一个专门的工具，使其能够“写入”长期记忆。

#### 新增工具: `save_user_fact`
- **目标**: 允许 Agent 提取并持久化关于用户的结构化事实。
- **触发时机**: 当 LLM 识别到用户提供了个人信息、偏好或约束条件时主动调用。

#### 工作流
1.  **用户输入**: “记住，我做后端开发喜欢用 Python。”
2.  **LLM 决策**: Agent 分析意图，调用 `save_user_fact(text="用户偏好使用 Python 进行后端开发")`。
3.  **后端执行**:
    - 工具接收文本。
    - 调用 Embedding 服务将文本向量化。
    - 将向量 + 元数据 (user_id, timestamp) 插入 Milvus 的 `facts` 集合。
    - **隔离性**: 插入操作严格绑定当前 `user_id`。

## 3. 图片存储与权限隔离架构

### 现状痛点
之前的图片处理仅生成了描述，未将图片作为可检索资产落地。我们需要一个健壮的存储方案，将重资产（文件）与元数据（向量）分离。同时，为了防止恶意攻击和资源滥用，必须引入严格的上传限制。

### 改造方案

#### 3.1 物理存储 (资产层)
- **位置**: 本地磁盘 `data/images/` (未来可无缝切换至 OSS)。
- **路径结构**: `{dept_id}/{user_id}/{timestamp}_{uuid}.jpg`
- **安全设计**:
    - **防路径穿越**: 严格校验 ID 格式。
    - **文件名处理**: 采用 `{timestamp}_{uuid}_{original_filename}` 格式。既保留原始文件名（用于检索和语义理解），又通过前缀保证唯一性和防止覆盖，同时需对原始文件名进行安全过滤（去除非法字符）。

#### 3.2 核心限制规则 (Constraints)
为保障系统稳定性与安全性，**前端与后端必须同时执行**以下限制：

| 限制项 | 阈值 | 说明 | 错误处理 |
| :--- | :--- | :--- | :--- |
| **单图大小** | **Max 10MB** | 防止带宽耗尽与内存溢出 (OOM) | 前端拦截提示 "图片过大"；后端返回 413 Payload Too Large |
| **文件名长度** | **Max 128 chars** | 防止文件系统异常及缓冲区溢出 | 截断或拒绝；建议后端自动截断保留后缀 |
| **文件类型** | `.jpg`, `.png`, `.jpeg`, `.webp` | 仅支持主流静态图像格式 | 拒绝非法 MIME Type |
| **分辨率** | **Max 4096 x 4096** | 防止超大分辨率导致 CV 模型推理崩溃 | 后端自动 Resize 至合理尺寸 (如 1024px) |

#### 3.3 Milvus 数据库设计 (元数据层)
Collection 名称: `agent_image_memory`

| 字段名 | 类型 | 维度/长度 | 描述 | 索引/备注 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | VARCHAR | 64 | 唯一主键 (UUID) | Primary Key |
| `dept_id` | VARCHAR | 64 | **部门ID** | Partition Key / 过滤核心 |
| `user_id` | VARCHAR | 64 | **用户ID** | 过滤核心 |
| `visibility` | VARCHAR | 20 | `private/department` | 权限过滤 |
| `image_uri` | VARCHAR | 1024 | 图片相对路径 | `dept_id/user_id/xxx.jpg` |
| `embedding` | FLOAT_VECTOR | 768 | CLIP 图像特征 | 向量索引 (IVF_FLAT/HNSW) |
| `content` | VARCHAR | 4096 | 图片描述/OCR文本 | 用于文本辅助检索 |
| `metadata` | JSON | - | 扩展信息 | 原始文件名、大小、MIME等 |
| `created_at` | INT64 | - | 创建时间戳 | 列表排序/审计 |
| `updated_at` | INT64 | - | 更新时间戳 | 更新追踪 |

#### 3.4 权限隔离规则 (Expr 过滤)
利用 Milvus 的布尔表达式 (Boolean Expressions) 实现高效的软隔离。

| 场景 | 角色 | 读权限 (Search) | 写权限 (Insert) |
| :--- | :--- | :--- | :--- |
| **默认/审计** | `default_user` / `default_dept` | **全局可见** (无过滤条件) | **拒绝 (Deny)** (硬拦截) |
| **普通员工** | `dept_1` 下的 `user_A` | **部门内可见** (`dept_id == 'dept_1'`) | **仅限本人本部门** (强制绑定 `user_A` & `dept_1`) |

#### 3.5 安全防御机制
1.  **空指针防御**: 代码层必须严格校验 `dept_id` 和 `user_id`，为空则拒绝操作。
2.  **并发安全**: CLIP 模型采用单例模式；文件 I/O 必要时加线程锁。
3.  **参数注入防御**: 使用参数化查询，防止注入攻击。
