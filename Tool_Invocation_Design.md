# Agent 工具调用全链路与可视化展示设计 (Tool Invocation & UI Design)

本文档详细阐述了在 LangGraph Agent 项目中，关于大模型“调用工具（Tool Calling）”全过程的前后端交互协议、参数传递机制以及前端可视化展示（思考过程）的架构设计方案。

## 一、 核心设计理念

1. **过程透明化 (Transparency)**：打破大模型的“黑盒”，让用户清晰看到 Agent 正在使用什么工具、传入了什么参数、得到了什么结果。这不仅能增加用户的信任感，也极大方便了开发期的 Debug。
2. **流式异步反馈 (Streaming Feedback)**：工具调用的生命周期（开始、参数接收、执行完毕）与大模型文本生成混合在同一个 SSE (Server-Sent Events) 流中，实现毫无卡顿的实时 UI 刷新。
3. **富媒体中间态展示**：不仅展示纯文本参数，针对图搜图等特殊工具，直接在“思考过程”中渲染出搜索到的图片缩略图，提供沉浸式的多模态交互体验。

---

## 二、 前后端通信协议 (SSE 流式事件设计)

为了区分“普通聊天文本”和“工具调用状态”，我们在标准的 OpenAI Stream Chunk 协议之上，扩展了自定义字段 `x_tool_event`。

### 数据包结构设计
后端在执行工具的不同阶段，会向前端推送携带 `x_tool_event` 的 JSON Chunk：

```json
// 1. 工具开始执行 (带参数)
data: {
  "x_tool_event": {
    "tool": "rag_image_search",
    "status": "started",
    "input": "{\"query\": \"蓝色帽子长发喜欢读书的女孩\", \"top_k\": 5}"
  }
}

// 2. 工具执行完毕 (带结果预览)
data: {
  "x_tool_event": {
    "tool": "rag_image_search",
    "status": "completed",
    "result_preview": "1. 马库斯.png (相似度: 0.893)\n2. 槲寄生小姐.png (相似度: 0.812)"
  }
}
```

---

## 三、 前端状态机与 UI 渲染架构

前端 (`script.js`) 通过拦截上述 SSE 数据流，构建了一个局部的“思考状态机”，用于驱动 UI 的动态更新。

### 1. 容器与文本隔离设计 (`thoughts-container`)
* **设计目标**：将大模型的“内心独白”（思考过程）与“最终回答”在视觉上隔离。
* **实现机制**：前端动态创建一个可折叠的 `<div class="thoughts-container">`。如果在调用工具前，大模型输出了一些过渡性废话（例如：“好的，我这就为您搜索...”），前端会**自动将这些零碎文本搬移到思考容器中**，保持最终 Markdown 内容的纯净。

### 2. 状态流转与参数安全展示 (Safe Parameter Rendering)
当接收到 `started` 事件时：
* 动态创建一个 `.status-item`，并打上 `data-tool="tool_name"` 标签。
* **参数安全解析**：对 `input` 字段进行安全尝试。若是 JSON 字符串则进行格式化 `JSON.stringify(input, null, 2)`；若超长（>100字符）则自动截断 `substring(0, 100) + '...'`，防止恶意或超大参数撑爆 UI。
* UI 呈现一个“旋转 Loading”图标，并显示“正在使用 xxx 工具... 参数：xxx”。

### 3. 结果回调与生命周期闭环
当接收到 `completed` 事件时：
* 前端通过 `data-tool` 属性，在 DOM 树中从下往上反向查找对应的未完成状态项。
* 将该状态项标记为 `.completed`，Loading 图标转换为绿色“完成 (Done)”对勾，文字更新为“已完成 xxx”。

## 四、 针对图搜图的富媒体特化设计 (Rich Media Result Preview)

通常的工具返回结果是一堆纯文本。但为了极致的用户体验，我们在前端对特定工具的 `result_preview` 做了“拦截与特化渲染”。

### 图像缩略图微图库 (Mini Image Gallery)
在 `x_tool_event` 触发 `completed` 且 `toolName === 'rag_image_search'` 时，前端不只是呆板地显示纯文本：

1. **正则捕获**：前端利用正则 `(\d+\.\s+)([^(\n]+\.(?:jpg|jpeg|png|gif|webp))` 实时扫描后端传来的文本结果（例如 `"1. 槲寄生小姐.png"`）。
2. **DOM 动态创建**：一旦匹配到图片文件名，前端会立即生成一个迷你的 `<img>` 标签，前缀拼上 `/assets/`，将其塞入到工具状态的下方。
3. **交互增强**：赋予缩略图 `height: 40px; cursor: pointer;` 等样式，并绑定 `onclick` 事件允许用户点击直接在新标签页查看大图。

### 业务价值
这一设计让用户在**大模型还在吭哧吭哧打字生成最终回答的时候**，就已经能从“思考过程”里一眼看到搜索出来的几张图片了，极大地缩短了用户的等待焦虑时间（Perceived Latency）。

---

## 五、 总结

这套设计将 **“工具调用”从后端的黑盒逻辑，变成了一个高度可观测、多模态、流式的前端互动组件**。它不仅处理了基础的文本参数解析与防溢出保护，还针对图像搜索业务做了深度定制，是提升 Agent 平台专业感与可用性的核心基石。
